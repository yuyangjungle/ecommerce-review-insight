# 技术架构与工作流

## 1. 系统概览

系统分为四层：

1. 数据层
2. 工作流层
3. 服务与 API 层
4. 展示层

## 2. 工作流设计

### 步骤 1：输入标准化

目标：将评论、问答、竞品信息统一为标准结构。

输出字段：

- source_type
- text
- rating
- metadata

### 步骤 2：主题抽取

目标：抽取评论中的高频主题与情绪倾向。

首版实现：

- 规则切分文本
- 按关键词或简单分类聚合
- 以结构化结果验证工作流与界面形态

后续版本：

- 接入真实 LLM 做主题命名与总结

### 步骤 3：洞察生成

基于主题与原始文本生成：

- 用户最关注问题
- 正向卖点
- 负向改进点
- FAQ 建议

### 步骤 4：引用关联

将每条洞察关联到 1 到 3 条代表性评论，降低“黑盒总结”风险。

### 步骤 5：提示词评估

同一份输入跑两套提示词工作流：

- v1：直接总结
- v2：先抽主题再总结

对比输出结构与可用性。

## 3. 推荐架构

### 前端

- 分析看板页面
- 数据集选择器
- 结果卡片
- 引用抽屉
- 提示词对比视图

### 后端

- `/health`
- `/datasets`
- `/analyze`
- `/evaluate`

### 存储

首版使用本地 JSON + SQLite。

## 4. 数据模型

### 评论

```json
{
  "product_id": "p001",
  "review_id": "r001",
  "rating": 5,
  "content": "续航很强，通勤一天完全够用。",
  "created_at": "2026-03-08"
}
```

### 洞察

```json
{
  "title": "用户认可续航表现",
  "type": "positive_theme",
  "summary": "多条评论提到日常通勤续航充足。",
  "citations": ["r001", "r008"]
}
```

### 生成结果

```json
{
  "asset_type": "faq",
  "title": "一天一充够吗？",
  "content": "根据多数评论，正常通勤场景基本可以满足一天使用。",
  "citations": ["r001", "r003"]
}
```

## 5. 评估方案

### 离线评估

- 结构完整度：是否包含主题、洞察、卖点、FAQ、引用
- 引用覆盖率：洞察是否附带引用
- 人工可用性：1 到 5 分
- 人工可信度：1 到 5 分

### 演示版评估

支持展示提示词 `v1` 与 `v2` 的输出差异。

## 6. 设计重点

当前架构重点强调：

- 先抽主题再生成，优先控制输出结构
- 为结论保留引用，提升结果可回溯性
- 评估不只关注生成流畅度，也关注结构和依据
- 首版以轻量、可解释方案验证交互流程与结果形态
