# 电商评论洞察助手 Case Study

## 1. 项目概述

### 项目名称

电商评论洞察助手

### 一句话介绍

帮助电商运营或商品经理从非结构化评论中快速提炼用户痛点、卖点机会、FAQ 草案与优化方向，并通过引用与提示词对比提升结果可解释性。

### 当前阶段

当前为：

`可交互本地 Demo / MVP 验证版`

## 2. 背景问题

在商品运营工作中，评论与问答是最重要的一手用户反馈来源，但它们存在三个典型问题：

1. 信息量大，人工通读成本高。
2. 表达分散，难以快速形成结构化判断。
3. AI 总结虽然快，但容易出现“看起来像对的，实际上没有依据”的问题。

因此，这个项目关注的不只是“能不能生成结论”，而是：

- 能不能把评论转成可执行的产品结论
- 能不能让结论可回溯、可验证
- 能不能比较不同 workflow 的质量差异

## 3. 目标用户

主要用户：

- 电商运营
- 商品经理
- 内容运营

次级用户：

- AI 产品经理
- 算法策略同学

## 4. 产品目标

这个项目希望帮助用户在短时间内完成以下任务：

1. 看懂评论里最重要的正负向主题。
2. 提炼用户最关心的问题。
3. 快速生成可用的卖点建议与 FAQ 草案。
4. 查看结论的原始依据，降低“AI 胡编”风险。
5. 对比不同提示词 workflow 的效果。

## 5. 方案设计

### 核心思路

我没有把它设计成一个泛泛的聊天机器人，而是拆成一条更产品化的 workflow：

1. 输入评论数据
2. 抽取主题
3. 提炼问题
4. 生成卖点与 FAQ
5. 附上引用
6. 对比 workflow 输出质量

### 为什么这么拆

- 先抽主题，再生成内容，更容易控制输出结构。
- 加引用回看，可以提升结果可信度与可解释性。
- 做 workflow 对比，而不是只给单一输出，有助于展示产品评估意识。

## 6. 当前版本实现了什么

### 前端

- 交互式分析页
- Executive Summary
- 原始评论样本区
- 主题、FAQ、卖点、优化方向卡片
- Prompt v1 / v2 对比区
- citation drawer

### 后端

- 本地 API 服务
- 样例数据读取
- 上传 JSON 数据分析
- 轻量规则抽取与 mock workflow
- 分析结果结构化返回

### 评估

当前版本展示了轻量评估维度：

- 引用覆盖率
- 结构完整度
- 可用性评分
- 可信度评分

## 7. 这个项目证明了什么

如果从 AI 产品岗视角看，这个项目主要证明：

1. 我能从一个业务场景出发定义问题，而不是只从模型能力出发找场景。
2. 我理解 AI 产品不只是“接个 API”，而是要设计 workflow、结果结构和解释机制。
3. 我知道评估不能只看生成是否流畅，还要看依据与可用性。
4. 我能把一个想法做成真实可运行的 demo，而不是停留在文档或原型层面。

## 8. 当前局限

当前版本还不是最终版主讲项目，主要局限有：

1. AI 能力层还是规则 + mock workflow，并未接真实 LLM。
2. 还没有公网部署。
3. 还没有真实用户反馈与系统化人工评测数据。
4. 上传格式目前仍偏工程化，不够消费级。

## 9. 下一步规划

为了把它升级成真正适合主讲的项目，下一步会优先做：

1. 接入真实 LLM provider
2. 增加更真实的 Prompt A/B 结果记录
3. 部署公网 demo
4. 补完整的 README 截图、录屏和 GitHub 展示材料

## 10. 适合怎么放进作品集

最推荐的呈现方式是：

1. 一页 case study
2. 一个 GitHub 仓库
3. 一个在线 demo

这三者的分工应该是：

- case study 负责讲清楚问题、方案和思考
- GitHub 负责提供代码与实现证据
- demo 负责让面试官快速建立直观印象

## 11. 面试口径示例

如果在面试里用 30 到 60 秒介绍，可以这样讲：

“这个项目是我围绕电商运营场景做的一个 AI 产品作品。我关注的问题是评论虽然很多，但人工提炼成本高，而 AI 总结又常常缺少依据。所以我把 workflow 拆成评论输入、主题抽取、问题提炼、内容生成、引用回看和提示词对比几个步骤。当前版本已经做成可交互本地 demo，可以上传评论并查看结果与引用。下一步我会接入真实 LLM 并做公网部署，把它升级成真正可公开展示的作品。”
