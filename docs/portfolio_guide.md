# 作品集呈现建议

## 1. 最推荐的提交组合

如果招聘系统支持多个链接，最推荐上传这三类材料：

1. GitHub 仓库
2. 在线 demo
3. 作品集 case study

如果只能放一个链接，优先放 GitHub，但要让 GitHub 首页成为总入口。

## 2. GitHub 应该怎么呈现

仓库首页最好具备：

1. 一句话介绍
2. 项目定位
3. 当前阶段说明
4. 核心功能
5. 本地运行方式
6. 文档入口
7. 后续路线图

换句话说，GitHub 不应该只是代码目录，而应该像一个“项目展示首页”。

## 3. 作品集页建议结构

如果你把内容放在 Notion、飞书文档或 PDF 中，建议结构如下：

1. 项目一句话介绍
2. 业务背景与用户问题
3. 目标用户
4. 方案拆解
5. 核心 workflow 图
6. 页面截图
7. 当前结果与评估方式
8. 当前局限与下一步

## 4. 最适合当前项目的叙事方式

当前这个项目最适合讲成：

`一个围绕电商评论洞察场景设计的 AI 产品 MVP，用来验证 workflow、解释性与评估闭环。`

不建议讲成：

- 已上线商用产品
- 已大规模落地系统
- 完整生产级 AI 平台

## 5. 截图建议

后续准备作品集截图时，建议至少保留这几张：

1. 首页总览
2. Executive Summary + 指标区
3. 主题与卖点区
4. Prompt 对比区
5. citation drawer 打开状态

## 6. 文案风格建议

作品集里的文案要避免两种极端：

1. 只写功能，不写为什么做
2. 只写愿景，不写你真正做了什么

更好的方式是每一段都包含：

- 问题
- 方案
- 结果或当前状态

## 7. 当前项目处于哪一步

按作品成熟度来看，可以分为：

1. 原型阶段
2. 可交互 demo 阶段
3. 公网可展示阶段
4. 真实上线阶段

当前项目处于：

`第 2 阶段：可交互 demo 阶段`

离第 3 阶段还差：

1. 真实 LLM
2. 公网部署
3. README 截图与录屏

## 8. 你现在最该优先补什么

优先级建议：

1. 接真实 LLM
2. 部署公网 demo
3. 补截图和短录屏
4. 再补更强的 README 首页视觉

## 9. 一句话结论

当前这个项目已经能作为作品集主项目雏形提交，但如果你希望它成为真正的“主讲项目”，最关键的升级是把 mock workflow 替换成真实 LLM，并完成公网展示。
